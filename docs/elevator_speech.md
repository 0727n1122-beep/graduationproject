# Elevator Speech — Team 10

## 프로젝트 개요

바이브 코딩을 활용하는 비전공자들이 LLM API 사용 시 발생하는 불필요한
토큰 낭비를 줄여주는 프롬프트 최적화 및 가이드 웹 서비스

**핵심 논리:**
> 비전공자를 위해, tiktoken과 규칙 기반 엔진 기술을 사용,
> 토큰 낭비 문제를 풀어서, AI를 똑똑하게 쓰게 한다.

---

## 발표 내용

### 문제 — 고객, Pain Point 

AI 없이는 뒤처지는 시대가 된 지금, 코딩에 문외한인 사람들도 AI를 이용해서
직접 개발을 할 수 있게 되었습니다.

그런데 이들 비전공자에겐 몇 가지 문제점이 있습니다. 토큰이 뭔지,
프롬프트를 어떻게 써야 하는지 모른 채로 사용한다는 것입니다.

코드 전체를 통째로 복사해서 "고쳐줘"라고 보내거나, 너무나 모호하고
포괄적인 요청을 하기도 합니다.

토큰은 시각적으로 확인할 수 없기 때문에, 결국 자기도 모르는 사이에
토큰을 수십 배 낭비하고 비용이 쌓입니다.

---

### 해결 아이디어

저희는 이 낭비를 눈에 보이게 만들고, 줄여주고, 스스로 고칠 수 있도록
만드는 플랫폼을 만들고 있습니다.

단순히 프롬프트를 고쳐주는 게 아니라, 비용적으로 개선하고 어디서 왜
낭비가 발생하는지 원리까지 보여주는 것이 핵심입니다.

---

### 기술/구현 

저희 프로젝트의 구현은 세 단계로 이루어집니다.

먼저 사용자가 프롬프트를 입력하면, 저희가 직접 설계한 규칙 기반 엔진이
한국어 filler word와 모호한 표현을 감지하고 제거합니다.

다음으로 토큰 수를 계산해 Before/After를 시각화하고, 어디서 왜 토큰이
낭비되는지 보여줍니다.

마지막으로 7개 LLM 모델의 예상 비용을 실시간으로 비교해 보여줍니다.

프론트엔드는 React, 백엔드는 FastAPI로 개발할 예정입니다.

---

### MVP/배포 

중간 발표 때는 예시 프롬프트 하나로 작동하는 프로토타입을 보여드렸습니다.

현재는 서비스의 전체 흐름과 UI가 완성된 상태이고, 기말까지는 실제 어떤
프롬프트를 넣어도 실시간으로 분석과 최적화가 작동하는 완성형 서비스로
만들 계획입니다.

배포는 Vercel을 사용해 웹 브라우저에서 누구나 바로 접속할 수 있게 할
예정입니다.

---

### 차별성 

기존 서비스들은 결과만 던져주거나, 여러 기능이 흩어져 있어 통합된 사용이
어렵습니다.

저희는 한국 비개발자를 타겟팅하여, 비개발자가 비효율적인 프롬프트를
요청하는 그 순간 프롬프트 개선, 비용 개선, 그리고 더 나아가 교육까지
통합합니다.

저희가 만드는 건 단순 최적화 도구가 아니라, 비개발자가 AI를 똑똑하게
쓰게 되는 경험입니다.

## PMF (Product-Market Fit)

### Problem-Solution Fit 검증

#### 문제의 실존 증명

**학술 연구 근거**

| 출처 | 내용 |
|------|------|
| CHI 2026 (Thorgeirsson et al.) | CS 지식이 낮을수록 바이브 코딩 성능 낮고 반복 프롬프트 증가 |
| METR (2025.07, Becker et al.) | 경험 많은 개발자도 AI 사용 시 19% 느려짐 → 비개발자는 더 심각 |
| Replit 연구 (Geng et al., 2025.07) | 초보자는 전체 문제를 한 번에 해결하려 함, 코드를 60%만 실행 |
| Grey Literature Review (arXiv:2510.00328) | "재프롬프트-붙여넣기 루프"에 빠지는 패턴 확인 |

**실무 데이터**

- 바이브 코딩 세션당 생성 코드의 30~40%가 폐기
- 비개발자 방식(전체 복붙)은 수만 토큰, 개발자 방식(필요 부분만)은 96% 절감

#### 솔루션의 효과

**프로토타입 검증**

- 예시 프롬프트 기준 약 40% 토큰 절감 확인
- 사용자가 "왜 비효율적인지"를 직관적으로 이해 가능
- Before/After 시각화로 개선 효과 명확

**핵심 가치**

| 가치 | 내용 |
|------|------|
| 시각화 | 보이지 않던 토큰 낭비를 실시간으로 표시 |
| 자동화 | 한국어 filler word와 모호 표현 자동 감지 |
| 교육 | 왜 비효율적인지 분석하고 개선 방법 제시 |

## 고객 인터뷰

### 인터뷰 개요

| 항목 | 내용 |
|------|------|
| 인터뷰 인원 | 2인 |
| 인터뷰 대상 | 바이브 코딩 경험이 있는 비전공자 |
| 인터뷰 방식 | 1:1 대면 인터뷰 |

---

### 인터뷰이 1

**배경**: 사회학/경영학 전공 대학생, 서비스 기획 진로 준비 중.
Claude, Gemini, GPT를 검색·과제·프로젝트 용도로 활발히 사용 중.

**주요 발견**

- 유료 결제 경험은 없으나 Claude, Lovable 등에서 토큰 소진 후 유료 결제 안내나
  장시간 대기를 요구받은 경험 있음. 대화가 중간에 끊기는 점이 가장 큰 불편.
- 프롬프트 작성 방식: "쇼핑몰 홈페이지, 카테고리별 페이지, 상세 페이지를 만들어줘"
  처럼 자연어로 기능을 설명하는 방식 사용. 결과 보며 수정 반복.
- 재요청 빈도가 높음. 원하는 점과 원하지 않는 점을 더 구체적으로 재입력하는 방식
  으로 대처.
- 토큰 개념을 정확히는 모르나 개념적으로는 이해(문장 처리 단위, 많을수록 비용 증가).
- 서비스 도입 의향: 긍정적. 특히 토큰 낭비 줄이기와 프롬프트 개선 기능이 매력적.
  실제 측정 방식이 궁금하다고 언급.

> "바이브코딩 관심자로서 이런 서비스의 필요성에 공감합니다."

---

### 인터뷰이 2

**배경**: 수학 전공, 학원 강사. 문제 제작·홍보에 AI 활용.
자체 교재 대량 제작을 위해 AI에게 코드 작성을 요청한 경험 있음.

**주요 발견**

- Claude 유료 플랜 사용 중. 몇 번 요청하면 5시간 사용 제한이 걸려 작업이
  반복적으로 중단되어 매우 불편함.
- 프롬프트 작성 방식: "문제를 입력하면 자동으로 시험지를 생성하는 코드를
  작성해 줘"처럼 자동화 코드를 요청. 오류 없는 결과물 완성까지 약 3일 소요.
- 오류 발생 시 어디가 문제인지 파악이 어려워 코드를 처음부터 전부 다시
  작성해달라고 요청하는 방식 주로 사용 → 전형적인 재프롬프팅 루프 패턴.
- 토큰 개념을 전혀 모름.
- 서비스 도입 의향: 긍정적. 월 3만 원 이상 지불 중임에도 30분 사용하면
  5시간 제한이 걸려 비용 대비 효율이 낮다고 느낌.

> "비용 대비 사용량을 최적화할 수 있다면 충분한 필요성을 느낍니다."

---

### 인터뷰 종합 분석

| 항목 | 인터뷰이 1 | 인터뷰이 2 |
|------|-----------|-----------|
| 토큰 개념 인지 | 부분적으로 앎 | 전혀 모름 |
| 재프롬프팅 빈도 | 높음 | 매우 높음 |
| 비용 불편 경험 | 있음 (무료 한도 소진) | 있음 (유료 사용에도 제한) |
| 서비스 도입 의향 | 긍정적 | 긍정적 |
| 특히 원하는 기능 | 토큰 낭비 분석, 프롬프트 개선 | 사용량 최적화, 비용 절감 |

**핵심 인사이트**

두 인터뷰이 모두 공통적으로 재프롬프팅 루프 패턴을 보였고, 토큰 개념
인지와 무관하게 비용 및 사용량에 대한 불편함을 명확히 경험하고 있었다.
특히 인터뷰이 2는 오류 발생 시 코드 전체를 다시 요청하는 전형적인
비개발자 패턴을 보여, 저희 서비스의 타겟과 정확히 일치한다.
